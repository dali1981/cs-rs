At the heart of it, you don’t have a “bug in one line.” You have an architectural boundary problem: execution/pricing code is encoding trade direction as a sign on net_cost, while accounting/reporting is sometimes treating that same field as “capital at risk / denominator” and sometimes as “cashflow,” and those are different concepts.

So the flow that needs addressing is the semantic pipeline from trade economics → stored fields → metrics, and right now it mixes three different meanings in one number:
	•	Cashflow at entry (credit/debit; can be negative/positive)
	•	Capital / margin requirement (must be non-negative; “locked” buying power)
	•	Return denominator (must be consistent with what you call “capital”)

What’s the architectural flow today (and why it breaks)

1) Execution layer decides economic sign
In execution, entry_pricing.net_cost is the canonical “what did I pay/receive” number:
	•	long straddle: positive debit
	•	short straddle: negative (credit)

That’s a cashflow sign convention. It’s totally valid.

2) Result model stores that cashflow in a field named like “cost”
StraddleResult.entry_debit stores the signed value (and the field name is misleading for shorts).

Now you have a single field that looks like “capital deployed,” but is actually “entry cashflow.”

3) Different consumers interpret it differently
	•	pnl_pct treats it as a magnitude of premium → uses .abs() → “return on premium magnitude”
	•	capital_required() treats it as capital deployed → uses signed raw value → “capital deployed can be negative”
	•	portfolio/summary uses capital_required() as the denominator for ROC and capital-weighted metrics → blow-up:
	•	negative total capital
	•	ROC sign flip
	•	guard triggers → weighted return forced to zero

So the architectural issue is: you don’t have a single, explicit “economic contract” for what each field means. You’re reusing a cashflow field as a capital field.

⸻

The “heart” fix: separate the three ledgers

You need a clean boundary that makes these concepts explicit and prevents accidental reuse.

A) Cashflow ledger (signed)

Represents money moving in/out of the account:
	•	entry_cashflow (credit positive or negative depending on your convention, but consistent)
	•	exit_cashflow
	•	fees_cashflow
	•	hedge_cashflows (stock trades)

This is what P&L reconciles with.

B) Exposure / capital-at-risk ledger (non-negative)

Represents “how much buying power is tied up”:
	•	capital_required (non-negative by definition)
	•	In Phase 1 you may approximate with premium magnitude for longs, but for shorts it should be margin/max-loss proxy.
	•	In your future design: BPR/IBKR margin engine outputs this.

C) Return denominator definition (explicit)

Represents which denominator you chose:
	•	return on premium magnitude
	•	return on capital required (BPR peak/avg)
	•	return on max loss
	•	etc.

Right now, your system implicitly uses:
	•	premium magnitude for pnl_pct
	•	signed cashflow for capital_required
	•	then mixes those in the same report.

⸻

The architectural flow you should enforce

1) Execution outputs cashflow + position description

Execution should output:
	•	legs, quantities, marks
	•	entry_cashflow (signed)
	•	optional: premium_magnitude = abs(entry_cashflow) (non-negative) if you want “return on premium” as a metric

But execution should not be the place that decides “capital required.” That’s an accounting/risk decision and may depend on the broker regime.

2) Accounting layer computes capital required from a dedicated engine

Accounting should compute:
	•	capital_required(trade_state, margin_mode) → non-negative

This is exactly where your MarginCalculator / BPR engine belongs. It should never be derived by “whatever sign net_cost had.”

3) Reporting layer chooses denominators consistently

Reporting should print:
	•	P&L (cashflow-based)
	•	Return on premium (if you want it): pnl / premium_magnitude
	•	Return on capital required (if you want it): pnl / peak_bpr
	•	Weighted returns should weight by non-negative capital only.

What to do right now vs correct long-term design

Immediate “make it coherent” patch (still not broker-correct)

You already see the minimal inconsistency:
	•	StraddleResult.capital_required() should be .abs() to match the rest.

That makes the report internally consistent (returns-on-premium-weighted), but it’s still economically wrong for short straddles because premium received is not capital tied up.

Correct architectural direction (what your new BPR plan is for)
	•	Keep entry_debit (rename later) as entry cashflow (signed)
	•	Add an explicit capital_basis (or margin_basis) object:
	•	premium_magnitude
	•	max_loss
	•	bpr_peak, bpr_avg
	•	Teach the report: “Return on capital” means “return on selected basis.”

That’s the “heart” change: the basis of capital must be computed independently from cashflow.
